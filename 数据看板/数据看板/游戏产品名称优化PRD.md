## 一、需求背景

### 1.1 业务背景

在薪酬数据总览中，**「产品名称」** 字段当前仅展示简短的游戏名称（如"和平精英"、"王者荣耀"），缺乏标准化的产品标识信息。而在活动创建模块中，**「游戏名称」** 下拉选项已经维护了一套包含**中文名称 + 英文缩写编码**的标准产品名称体系，例如：

- 王者荣耀(WZRY)
- 和平精英(HPJY)
- 地下城与勇士手游(DXCYYSSY)
- 原神(YS)
- 英雄联盟手游(YXLMSY)
- 金铲铲之战(JCCZ)
- 元梦之星(YMZX)
- 英雄联盟(YXLM)

当前存在的问题：

1. **产品名称不标准**：薪酬数据总览中的「产品名称」仅为简称（如"和平精英"），缺少英文缩写编码，无法与活动创建模块中的标准名称对齐
2. **跨系统识别困难**：在与上下游系统（如财务系统、立项系统）对接时，缺乏统一的产品编码标识，导致人工核对效率低
3. **数据追溯不便**：运营人员在薪酬数据总览中看到"英雄联盟"无法快速区分是端游(YXLM)还是手游(YXLMSY)，增加了理解成本
4. **与活动创建数据不一致**：活动创建时选择的是标准游戏名称（含编码），但到薪酬数据总览展示时丢失了编码信息

### 1.2 需求目标

将薪酬数据总览中的 **「产品名称」** 字段，替换为活动创建模块中 **「游戏名称」** 下拉选项里的标准化产品名称（含英文缩写编码），实现产品名称在全链路的标准化和一致性。

**核心目标：**

- **统一产品标识**：薪酬数据总览的「产品名称」与活动创建的「游戏名称」保持完全一致
- **提升辨识度**：产品名称包含英文缩写编码，便于快速识别和跨系统比对
- **消除歧义**：通过编码区分同名但不同产品类型的游戏（如端游 vs 手游）

### 1.3 目标用户

| 用户角色 | 当前痛点 | 优化后收益 |
|---------|---------|-----------|
| **运营人员** | 产品名称过于简略，无法快速区分端游/手游 | 标准化名称一目了然，减少误判 |
| **财务人员** | 与财务系统核对时需要额外查找产品编码 | 产品编码直接展示，提高核对效率 |
| **项目负责人** | 跨团队沟通时产品名称不统一 | 全链路统一标准名称，减少沟通成本 |

## 二、产品方案

### 2.1 方案概述

将薪酬数据总览表格中第4列 **「产品名称」** 字段的数据值，从当前的简短游戏名称替换为**活动创建模块「游戏名称」下拉选项中的标准化名称**，格式为 **「中文名称(英文缩写)」**。

## 三、功能详细描述

### 3.1 薪酬数据总览 - 产品名称展示

#### 3.1.1 展示规则

| 规则项 | 说明 |
|--------|------|
| **字段位置** | 数据表格第4列「产品名称」 |
| **展示格式** | `中文名称(英文缩写)`，如 `王者荣耀(WZRY)` |
| **编码括号** | 使用半角小括号 `()`，编码全大写 |
| **列宽调整** | 因名称变长（如"地下城与勇士手游(DXCYYSSY)"），需适当增加该列的默认宽度，建议从原来的 100px 调整为 **180px** |
| **超长处理** | 文本超出列宽时显示省略号 `...`，鼠标悬浮(hover)显示完整内容 |

#### 3.1.2 筛选影响

| 筛选维度 | 变更说明 |
|---------|---------|
| **全局搜索** | 支持按新格式搜索，如输入"WZRY"可命中"王者荣耀(WZRY)"的数据行 |
| **COA名称筛选** | 不受影响，COA名称维持原有逻辑 |
| **产品名称筛选**（如后续新增） | 筛选选项值需同步为标准化名称格式 |

#### 3.1.3 排序影响

- 产品名称列排序时，按中文部分的拼音首字母排序
- 英文缩写编码不影响排序逻辑

#### 3.1.4 导出影响

| 场景 | 说明 |
|------|------|
| **Excel导出** | 导出数据中「产品名称」列值为标准化名称（含编码），如"王者荣耀(WZRY)" |
| **来源说明行** | 产品名称的来源说明更新为：`[活动创建] 取自活动创建模块的「游戏名称」下拉选择值` |

### 3.2 自由字段展示（未映射字段）

#### 3.2.1 需求说明

薪酬数据总览共有 **51个字段**，其中部分字段已有明确的数据来源映射关系（如薪酬导入、活动创建、公式计算等），但仍有一部分字段 **暂未建立映射关系**（即"自由字段"）。

当前问题：这些未映射的自由字段在数据看板中被统一显示为 `-`（破折号），导致即使数据库中已有实际数据值，也无法在看板中展示给用户。

> 💡 **核心原则**：不论字段是否已建立数据来源映射，只要该字段有实际数据值，就应该在薪酬数据总览中正常展示，不应因为来源未定义而隐藏数据。

#### 3.2.2 自由字段列表

以下字段当前标记为"无来源"（type=none），属于自由字段范畴：

| # | 字段名称 | 当前展示 | 优化后展示 | 说明 |
|---|---------|---------|-----------|------|
| 1 | **系统来源** | `-` | 有值则展示原值（如"BFC"） | 系统写入的来源标记 |
| 2 | **产品ID** | `-` | 有值则展示原值（如"9768"） | 关联产品库的ID |
| 3 | **产品名称** | `-` | 按标准名称展示（本PRD主要需求） | 见3.1节 |
| 4 | **SG** | `-` | 有值则展示原值（如"国内发行及二方"） | 业务分组 |
| 5 | **产品类型** | `-` | 有值则展示原值（如"次新品"） | 产品分类 |
| 6 | **验收状态** | `-` | 有值则展示原值（带状态标签样式） | 流程状态字段 |
| 7 | **验收单号** | `-` | 有值则展示原值 | 系统自动生成 |
| 8 | **立项单号** | `-` | 有值则展示原值 | 关联立项系统 |
| 9 | **立项名称** | `-` | 有值则展示原值 | 关联立项系统 |
| 10 | **采购类别** | `-` | 有值则展示原值（如"内容营销"） | 采购分类 |
| 11 | **采买类型** | `-` | 有值则展示原值（如"视频图文"） | 采买内容类型 |
| 12 | **场景** | `-` | 有值则展示原值（如"KOL采买"） | 使用场景 |
| 13 | **合作模式** | `-` | 有值则展示原值（如"商单采买"） | 合作方式 |
| 14 | **采买方式** | `-` | 有值则展示原值（如"购买"/"配送"） | 采购方式 |
| 15 | **原平台** | `-` | 有值则展示原值 | 关联平台信息 |
| 16 | **游可爱id** | `-` | 有值则展示原值 | 备注信息（含刊例价等） |
| 17 | **账号分级** | `-` | 有值则展示原值（如"S级"/"A级"） | 账号等级 |
| 18 | **账号圈层** | `-` | 有值则展示原值（如"游戏圈"） | 账号所属圈层 |
| 19 | **账号粉丝数** | `-` | 有值则展示原值（如"1230万"） | 粉丝数量 |
| 20 | **已执行总价（不含服垫税减）** | `-` | 有值则展示原值（千分位格式化） | 未含税的执行总价 |

#### 3.2.3 展示规则

| 规则项 | 说明 |
|--------|------|
| **有数据时** | 正常展示数据原值，使用灰色文本（`color: #6b7280`）以区分于已映射字段 |
| **无数据时** | 展示灰色破折号 `—` |
| **hover提示** | 鼠标悬浮时tooltip显示 `[自由字段] 字段名称`，提示该字段暂未建立数据来源映射 |
| **全局搜索** | 自由字段中的数据值也需要纳入全局搜索范围 |
| **排序** | 自由字段支持正常排序 |
| **导出** | 导出Excel时，自由字段有数据则导出原值，无数据则为空 |

> ⚠️ **注意**：自由字段展示为**过渡方案**。随着系统迭代，这些字段会逐步建立明确的数据来源映射关系。建立映射后，字段将从"自由字段"升级为"已映射字段"，文本颜色恢复为正常色。

### 3.3 活动创建模块 - 游戏名称选择（现有功能，无需改动）

活动创建模块的「游戏名称」下拉选择框为本次需求的数据源头，**无需功能改动**，仅需确认：

| 确认项 | 说明 |
|--------|------|
| **下拉选项** | 维持现有的标准游戏名称列表（见2.2映射表） |
| **数据存储** | 活动记录中需完整存储所选游戏名称的标准值（含编码） |
| **扩展性** | 后续新增游戏产品时，在游戏名称下拉列表中新增即可，薪酬数据总览自动关联 |

### 3.4 存量数据处理

#### 3.4.1 历史数据迁移

对于需求上线前已存在的薪酬数据，需要进行一次性的数据迁移：

| # | 迁移规则 | 说明 |
|---|---------|------|
| 1 | **精确匹配** | 旧产品名称与标准名称的中文部分精确匹配时，自动补全编码。如"和平精英" → "和平精英(HPJY)" |
| 2 | **模糊匹配** | 旧名称为简称/别名时，需人工维护映射关系。如"DNF手游" → "地下城与勇士手游(DXCYYSSY)" |
| 3 | **无法匹配** | 若旧名称无法映射到标准名称，保持原值不变，标记为"待确认"，由运营人工处理 |
| 4 | **迁移时机** | 功能上线前完成全量历史数据迁移，上线后新数据自动使用标准名称 |

## 四、数据关联与技术要求

### 4.1 数据关联逻辑

```
┌─────────────────────────────────────────────────────────────┐
│                     产品名称数据流                            │
│                                                             │
│  游戏产品库（标准数据源）                                      │
│  ┌──────────────────────────────────────────┐               │
│  │ 产品ID │ 中文名称        │ 英文缩写    │ 标准展示值             │
│  │ 10001  │ 王者荣耀        │ WZRY       │ 王者荣耀(WZRY)        │
│  │ 10002  │ 和平精英        │ HPJY       │ 和平精英(HPJY)        │
│  │ 10003  │ 地下城与勇士手游 │ DXCYYSSY   │ 地下城与勇士手游(DXCYYSSY)│
│  │ ...    │ ...            │ ...        │ ...                  │
│  └──────────────────────────────────────────┘               │
│       ↓                          ↓                          │
│  活动创建模块                 薪酬数据总览                      │
│  (游戏名称下拉选择)           (产品名称字段展示)                  │
│                                                             │
│  两端共用同一产品库数据，保证名称一致性                           │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 技术实现要点

| # | 要点 | 说明 |
|---|------|------|
| 1 | **统一产品库** | 建立/复用游戏产品主数据表，包含产品ID、中文名称、英文缩写、标准展示值等字段 |
| 2 | **活动创建关联** | 活动创建时选择游戏名称后，将产品ID写入活动记录 |
| 3 | **薪酬总览取值** | 薪酬数据总览通过关联活动记录的产品ID，从产品库获取标准展示值 |
| 4 | **缓存策略** | 产品库数据变化频率低，建议前端缓存产品映射，减少查询压力 |
| 5 | **扩展新增** | 新增游戏产品只需在产品库中新增记录，活动创建和薪酬总览自动生效 |

### 4.3 接口变更（建议）

| 接口 | 变更说明 |
|------|---------|
| **获取产品列表** | 返回字段增加 `displayName`（标准展示值），格式为 `中文名称(英文缩写)` |
| **薪酬数据总览查询** | 返回的 `productName` 字段值变更为标准展示值 |
| **活动创建提交** | 无变更，维持现有的产品ID关联逻辑 |
