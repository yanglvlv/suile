## 一、需求背景

### 1.1 业务背景

公司直播业务线涵盖多款游戏（天龙八部手游、FC ONLINE、DNF系列、石器时代、天堂2等），每款游戏的主播激励政策规则各不相同，且会随运营节奏频繁调整。

目前存在以下问题：

1. **人工计算容易出错**：依赖人工手动计算结算金额，计算逻辑复杂且缺乏校验机制，极易产生数据错误

### 1.2 需求目标

构建一套**低代码政策配置中台**，实现：

- **零开发配置**：运营/产品人员可通过可视化界面自主完成政策规则的全流程配置
- **100%规则覆盖**：现支持直播业务10种政策类型，9种算子组件 + 通用条件组 + 无限嵌套，覆盖全部配置需求
- **自动化结算**：配置完成 → 提交审核 → 负责人审批 → 导入数据 → 引擎核算 → 生成结算单 → 导出，端到端自动化
- **数据可溯源**：每条结算明细支持算子执行链路溯源，支持在线修改和修改记录追踪

### 1.3 目标用户

| 用户角色 | 使用场景 | 核心诉求 |
|---------|---------|---------|
| **运营人员** | 配置/调整政策规则、导入数据、提交审核、执行核算、导出结算单 | 快速上手、灵活组装、准确高效、异常可追溯、支持修改 |
| **部门负责人** | 审核下属员工提交的活动配置，检查政策规则的合理性 | 快速审批、配置透明、审核可追溯 |
| **技术开发** | 维护算子引擎、扩展新算子 | 数据驱动、模型清晰、易扩展 |


### 2.2 核心设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **一切皆条件组** | 所有筛选/准入/分级逻辑统一用「通用条件组」表达 |
| 2 | **运算符全覆盖** | 比较运算符8种（`>`, `>=`, `<`, `<=`, `=`, `!=`, `BETWEEN`, `IN`），逻辑运算符3种（AND/OR/NOT） |
| 3 | **无限嵌套** | 条件组支持递归嵌套子条件组，表达任意复杂逻辑 |
| 4 | **一切皆可增删** | 条件行、档位行、政策Tab全部支持动态增删排序 |
| 5 | **数据驱动渲染** | 所有UI由JSON数据模型驱动，改数据即改界面 |

## 三、功能详细描述

### 3.1 活动基础信息

#### 3.1.1 功能说明

定义当前活动的基本参数，是政策配置的入口。

#### 3.1.2 配置项

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 活动名称 | 下拉选择 | 是 | 获取活动列表中正在进行中的活动 |
| 开始时间 | 日期展示 | — | 选择活动后自动获取该活动的开始时间，仅展示不允许修改 |
| 结束时间 | 日期展示 | — | 选择活动后自动获取该活动的结束时间，仅展示不允许修改 |
| 付款方式 | 下拉选择 | 是 | 选项：个人直发 / 机构代付 |
| 达人确认 | 开关(Switch) | 否 | 默认关闭。开启后，上级审核通过需先由达人在小程序确认金额后再推送财务内审；关闭则上级审核通过后直接推送财务内审 |
| 确认/票据截止时间 | 日期选择 | 条件必填 | 仅在"达人确认"开启时显示且必填。达人需在此时间前完成金额确认和票据上传，超时系统自动提醒 |

#### 3.1.3 业务规则

- 活动名称下拉列表仅展示状态为"进行中"的活动
- 选择活动后，开始时间和结束时间自动填充为该活动的时间范围，不允许手动修改
- 活动时间为全局默认时间，各政策可在属性面板中独立覆盖
- 「达人确认」开关控制结算流程分支：开启 = 需达人确认后再内审；关闭 = 直接内审
- 「确认/票据截止时间」仅在达人确认开启时可见，设置后对达人确认和票据上传共用同一截止时间

#### 3.1.4 活动状态机

活动全生命周期根据「达人确认」开关的状态分为两条流程分支：

**流程A：开启达人确认**

```
草稿 → 提交审核 → 待上级审核 → 上级审核通过 → 待达人确认 → 达人确认完成 → 财务内审 → 内审通过 → 待上传票据 → 票据齐全 → 推送FPP → 已完成
                    │                                                    │
                    └── 上级驳回 → 已驳回(退回修改)                        └── 财务驳回 → 已驳回(退回修改)
```

**流程B：关闭达人确认（默认）**

```
草稿 → 提交审核 → 待上级审核 → 上级审核通过 → 财务内审 → 内审通过 → 待上传票据 → 票据齐全 → 推送FPP → 已完成
                    │                           │
                    └── 上级驳回 → 已驳回         └── 财务驳回 → 已驳回(退回修改)
```

**状态说明**

| 状态 | 可见角色 | 可执行操作 |
|------|---------|-----------|
| **草稿** | 创建者 | 编辑、保存、提交审核 |
| **待上级审核** | 创建者(只读) + 审核人 | 审核人：通过 / 驳回 |
| **已驳回** | 创建者 | 查看驳回原因、修改配置、重新提交审核 |
| **待达人确认** | 创建者(只读) + 达人(小程序) | 达人：确认金额 |
| **财务内审中** | 创建者(只读) + 财务 | 财务：通过 / 驳回 |
| **待上传票据** | 创建者(只读) + 达人(小程序) | 达人：上传发票、完税证明 |
| **已推送FPP** | 所有人 | 查看结算单、票据 |
| **已完成** | 所有人 | 只读查看 |

### 3.2 数据导入与字段标准化

#### 3.2.1 功能说明

上传结算数据表，系统自动完成字段映射、数据校验和质量检测，为后续引擎核算提供标准化数据源。

#### 3.2.2 功能流程

```
上传数据文件 → 解析表头 → 自动字段映射 → 映射结果确认 → 数据预览 → 异常检测 → 完成
```

#### 3.2.3 详细功能

| 功能点 | 说明 |
|--------|------|
| **文件上传** | 支持 `.xlsx` / `.csv` 格式 |
| **自动映射** | 系统自动将导入列名与标准字段库（80+标准字段）比对 |
| **映射策略** | ① 精准命中（列名完全匹配）→ ② 别名纠偏（相似度≥50%自动匹配）→ ③ 待确认（需手动选择） |
| **推荐候选** | 未匹配的列推荐Top3最相似的标准字段，支持一键采纳 |
| **数据预览** | 展示前10行导入数据 |
| **异常检测** | 检测空值、类型错误（数值列含非数字）、负数等异常 |
| **统计摘要** | 展示数据行数、精准命中数、别名纠偏数、待确认数、异常行数 |

#### 3.2.4 标准字段库

系统内置80+标准字段，覆盖以下维度：

| 维度 | 包含字段 |
|------|---------|
| **流量指标** | 场观UV、进直播间人数、进直播间人数UV、日均UV、月场观UV |
| **时长指标** | 直播总时长(h)、有效天数、周时长(h)、累计时长(h)、开播时长(h) |
| **互动指标** | 总互动量、点赞数、转发数、评论数 |
| **商业指标** | 已执行总价、付费金额、流水(元)、CPM |
| **主播属性** | 机构ID、主播等级、是否新主播、招募渠道、签约月份数 |
| **游戏指标** | 赛季段位、赛季积分、组队完成场次、全服分数、击倒分数、战盟段位 |
| **平均指标** | 平均在线人数ACU、日均ACU、场均UV、单场最高在线 |
| **计算指标** | 影响力积分、ACU增幅比例、场观溢出比例 |

#### 3.2.5 业务规则

- 相似度算法采用编辑距离 + 关键词包含 + 归一化对比的综合评分
- 映射完成后用户可手动调整映射关系
- 重传文件会清空之前的映射结果

### 3.3 政策画布（核心功能）

#### 3.3.1 功能说明

每个活动可包含**N个政策**，每个政策由若干算子节点组装而成。政策通过Tab页切换，算子节点在画布区域通过**拖拽/点击**添加，支持拖拽排序。

#### 3.3.2 政策属性面板

位于画布右侧的属性面板，用于配置当前选中政策的全局属性。

| 配置项 | 字段名 | 类型 | 必填 | 默认值 | 说明 |
|--------|--------|------|------|--------|------|
| **政策名称** | `name` | 文本 | 是 | — | 政策标签页上显示的名称 |
| **活动时间** | `activityTimeStart` / `activityTimeEnd` | 日期 | 否 | 继承活动时间 | 每个政策可独立设定考核时间范围，留空则使用活动整体时间 |
| **合并关系** | `relation` | 下拉 | 是 | `add` | 决定该政策与其他政策金额的合并方式 |
| **互斥组成员** | `mutexGroup` | 多选 | 条件 | — | 当合并关系为"互斥"时显示，勾选哪些政策与当前政策互斥 |
| **有效天数定义** | `validDayHours` | 数字 | 否 | 1 | N小时=1个有效天，留空/0使用全局默认(1小时=1天) |

##### 合并关系说明

| 值 | 名称 | 说明 |
|-----|------|------|
| `add` | 叠加 | 各政策金额累加（**默认**） |
| `max` | 取最高 | 多政策取最高金额，只发最多的那个 |
| `mutex` | 互斥多选一 | 与指定政策组互斥，主播只能选一个领取 |

##### 互斥组交互规则

- 选择"互斥多选一"后展示互斥组成员勾选面板
- 勾选互斥成员后**自动双向同步**：对方政策也被标记为互斥关系
- 取消互斥后，若互斥组内无其他成员，**自动回退为叠加**(`add`)
- 政策Tab上展示互斥标识

#### 3.3.3 政策操作

| 操作 | 说明 |
|------|------|
| **添加政策** | 点击标签栏的"+ 加政策"按钮 |
| **复制政策** | 右侧属性面板底部的"复制政策"按钮，完整复制所有算子配置 |
| **删除政策** | 右侧属性面板底部的"删除政策"按钮；点击后弹出二次确认弹窗，确认后执行删除（至少保留一个政策） |
| **切换政策** | 点击Tab标签页切换 |

#### 3.3.4 画布操作

| 操作 | 说明 |
|------|------|
| **拖拽添加** | 从左侧算子面板拖拽到画布区域添加算子节点 |
| **点击添加** | 直接点击左侧算子面板中的算子 |
| **拖拽排序** | 在画布中拖拽节点卡片调整算子执行顺序 |
| **删除算子** | 点击节点右上角的 × 按钮 |
| **收起/展开面板** | 左侧算子面板和右侧属性面板均可独立收起 |

### 3.4 通用条件组（核心复用组件）

#### 3.4.1 功能说明

通用条件组是政策配置中**最核心的基础组件**，被多个算子复用：准入/筛选条件、排名每档准入条件、固定金额准入条件、奖池平分达标条件等。

#### 3.4.2 条件组结构

条件组支持三种逻辑运算符，通过下拉选择框切换（默认 `AND`）：

| 逻辑运算符 | 标签 | 说明 | UI样式 |
|------------|------|------|--------|
| `AND` | 且(全部满足) | 所有子条件都必须为真 | 绿色标签 |
| `OR` | 或(任一满足) | 任一子条件为真即可 | 黄色标签 |
| `NOT` | 非(取反) | 通常只包含一个子条件/组 | 红色标签 |

条件组内支持两种元素：

| 元素类型 | 说明 |
|---------|------|
| **简单条件行** | 字段 + 运算符 + 值，是最小的条件单元 |
| **嵌套子条件组** | 递归嵌套，支持**无限层级**，可表达任意复杂逻辑 |

#### 3.4.3 简单条件行

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `field` | 下拉 | — | 从标准字段库中选择（80+字段）+ "自定义..." |
| `operator` | 下拉 | `>=` | 比较运算符（8种） |
| `value` | 输入 | — | 比较值 |
| `value2` | 输入 | — | 仅BETWEEN时显示，作为区间上限 |

#### 3.4.4 比较运算符

| 运算符 | 标签 | 值类型 | UI表现 |
|--------|------|--------|--------|
| `>` | 大于 | 单值 | 单输入框 |
| `>=` | 大于等于 | 单值 | 单输入框 |
| `<` | 小于 | 单值 | 单输入框 |
| `<=` | 小于等于 | 单值 | 单输入框 |
| `=` | 等于 | 单值 | 单输入框 |
| `!=` | 不等于 | 单值 | 单输入框 |
| `BETWEEN` | 区间 | 双值 | 两个输入框（下限~上限） |
| `IN` | 属于列表 | 多值 | 逗号分隔输入 |

#### 3.4.5 交互设计

- 逻辑运算符通过下拉选择框切换 AND / OR / NOT
- "+ 条件"按钮添加简单条件行
- "+ 子组"按钮添加嵌套子条件组
- 每行/每组右侧有删除按钮
- 嵌套层级通过背景色深度递增来区分

### 3.5 算子组件详细说明

系统提供 **9种算子组件**，分为三大类：

| 分类 | 算子 | 用途 |
|------|------|------|
| **前置条件类** | ① 准入/筛选条件、② 计算字段定义 | 数据预处理，决定谁有资格、生成新指标 |
| **核心规则类** | ③ 阶梯排名发奖、④ 阶梯达标/溢出即发、⑤ 固定金额即发、⑥ 固定奖池平分、⑦ 流水分成/返点、⑧ 裂变伯乐激励 | 核心奖励计算逻辑 |
| **直发类** | ⑨ 表格直发 | 按名单直接发放 |

---

#### 3.5.1 算子① — 准入/筛选条件（condition-filter）

**用途：** 设置主播参与结算的前置资格条件。只有满足所有条件的主播才能进入后续的排名/瓜分/发奖环节。

##### 基础配置

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 名称 | `name` | 文本 | — | 过滤器名称 |
| 条件组 | `conditionGroup` | 条件组 | — | 使用通用条件组（AND/OR/NOT嵌套） |
| 未达标处置 | `failAction` | 下拉 | `eliminate` | 淘汰 / 折算 |

##### 未达标处置方式详细说明

| 值 | 名称 | 说明 | 展开配置 |
|-----|------|------|----------|
| `eliminate` | 淘汰 | 不满足条件的主播**直接排除**，不参与后续任何排名/瓜分/发奖环节，结算金额为0 | 无额外配置 |
| `discount` | 折算 | 不满足条件的主播**仍然参与结算**，但按完成率打折发放奖励。系统根据折算配置（基础折算方式+豁免规则+附加规则）计算出折算系数（0~1），最终金额 = 原始金额 × 折算系数 | 展开折算配置面板（三层结构） |


##### 折算配置（三层结构，failAction=discount时展开）

**第一层：基础折算方式**

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 折算方式 | `discountMethod` | 下拉 | `bucket_min` | 选择折算系数的计算方法 |

| 方式 | 说明 | 计算公式 | 额外配置 |
|------|------|----------|----------|
| `bucket_min` | 木桶极小值：多维度取最低完成率 | 系数 = Min(各维度完成率)。如配置了时长(目标60h)和天数(目标15天)两个分母，则系数 = Min(实际时长/60, 实际天数/15) | 折算分母列表（字段+目标值），支持多个维度 |
| `ratio` | 按比例折算：单一维度完成率 | 系数 = 该维度实际值/目标值。只看一个指定维度的完成比例，如配置折算维度为时长(目标60h)，则系数 = 实际时长/60 | 折算分母列表（字段+目标值），**限制只能配置1个维度**（添加1个后"添加维度"按钮置灰）。若需多维度折算请选择 `bucket_min` 或 `weighted_avg` |
| `fixed` | 固定系数：不达标直接按固定比例发放 | 系数 = 固定值（如0.8表示打8折） | 固定折算系数（0~1） |
| `weighted_avg` | 加权平均：多维度按权重加权 | 系数 = Σ(各维度完成率 × 权重)。如时长权重0.6、天数权重0.4，则系数 = (实际时长/目标时长)×0.6 + (实际天数/目标天数)×0.4 | 折算分母列表（字段+目标值+权重） |

**第二层：豁免规则（可选，所有折算方式通用）**

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 启用豁免 | `enableExemption` | 开关 | 关闭 | 开启后指定某维度达标即跳过所有折算，系数直接为1 |
| 豁免维度 | `exemptField` | 下拉 | — | 从标准字段库选择豁免判断的维度字段 |
| 目标值 | `exemptTarget` | 数字 | — | 豁免维度的达标目标值 |

> **豁免逻辑：** 豁免维度达标 → 系数=1（全额发放）；豁免维度未达标 → 走第一层基础折算方式计算系数

**第三层：附加规则（所有折算方式通用）**

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 最低折算比例 | `minDiscountRatio` | 数字 | 0 | 折算系数低于此值则该主播不予结算（直接淘汰）。0表示不限制 |

#### 3.5.2 算子② — 计算字段定义（computed-field）

**用途：** 用已有字段通过公式计算出新虚拟字段，供后续排名或达标判断使用。

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 字段名 | `fieldName` | 文本 | — | 新字段名称 |
| 计算公式 | `formula` | 文本 | — | 自定义计算公式，支持引用标准字段名和 +、-、×、÷、()。如 `点赞数 * 0.2 + 转发数 * 0.5 + 评论数 * 1`、`付费金额 / 10` |

**公式输入辅助**：
- 提供"插入字段"下拉按钮，运营可从标准字段列表中点选插入，避免手动输入字段名拼写错误
- 提供运算符快捷按钮（+、-、*、/、(、)），点击自动插入到公式光标位置
- 实时校验：公式中引用的字段名如果不在标准字段列表中，会显示红色警告提示；全部匹配则显示绿色通过

---

#### 3.5.3 算子③ — 阶梯排名发奖（rank-rule）

**用途：** 按指标排名分档发放奖励。是使用频率最高的核心算子。支持单指标排名、多维加权排名。每档可设独立准入条件+名额限制。

##### 基础配置

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 排名方式 | `sortMethod` | 下拉 | `single` | `single`(单指标) / `weighted`(多维加权)。如需按自定义公式排名，可前置「计算字段定义」算子生成新字段，再用 `single` 排名 |
| 排序配置 | `sortFields` | 数组 | — | 多级排序，每级含字段+排序方向(desc/asc) |
| 空额累积 | `quotaOverflow` | 开关 | `none` | `cascade`(高档位未满员→名额累积到低一档) / `none`(不累积) |
| 金额模式 | `amountMode` | 下拉 | `fixed` | `fixed`(固定总额) / `unit_price`(单价×时长) |

##### 多维加权配置（sortMethod=weighted时）

| 配置项 | 说明 |
|--------|------|
| 权重配置 | N个维度字段 + 权重百分比（总和=100%） |

> **注**：若某维度未达标需惩罚（如打折发放），由前置的「准入/筛选条件」算子的**折算（discount）**功能实现，不在排名算子中配置。

##### 档位表（动态增删行）

| 属性 | 说明 |
|------|------|
| `tiers[].name` | 档名 |
| `tiers[].amount` | 金额（数值） |
| `tiers[].limit` | 该档位名额限制（留空不限） |
| `tiers[].condGroup` | 每档独立准入条件（复用通用条件组） |
| `tiers[].condNote` | 档位条件备注说明 |

##### 特殊说明字段

| 字段 | 说明 |
|------|------|
| `rankNote` | 排名规则说明卡片，展示特殊规则 |
| `exclusionNote` | 互斥排他说明卡片 |
| `quotaOverflowNote` | 空额累积规则备注 |

---

#### 3.5.4 算子④ — 阶梯达标/溢出即发（target-rule）

**用途：** 按指标阈值分档，达到哪档发哪档金额（不排名，达标即发）。支持绝对值和百分比（相对基线）两种指标形态。每档可设独立准入条件。

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 对标字段 | `targetField` | 下拉 | — | 用于阈值判断的字段（从标准字段库选择），如"月均ACU"、"场观UV" |
| 指标形态 | `metricType` | 下拉 | `absolute` | `absolute`(绝对值) / `percentage`(百分比·相对基线) |
| 基线字段 | `baseField` | 下拉 | — | 百分比模式时的基准字段（metricType=percentage时必填） |
| 基线值 | `baseVal` | 数字 | — | 百分比模式时的基准值（metricType=percentage时必填） |
| 规则说明 | `ruleNote` | 文本 | — | 特殊规则备注卡片 |

**指标形态说明**

| 值 | 说明 | 公式 | 阈值单位 | 正负 |
|-----|------|------|----------|------|
| `absolute` | 直接用字段原始值判断 | 无 | 与字段原始单位一致 | 仅正值 |
| `percentage` | 相对基线的偏移百分比 | (实际值 - 基线值) / 基线值 × 100% | **百分比（%）** | **支持正负**（正=超出/增长，负=低于/下降） |

> ⚠️ **百分比模式的阈值为百分比数值**：例如填写 `60` 表示 60%，填写 `-30` 表示 -30%。系统在展示时会自动加 `%` 后缀。适用于溢出比例、增长率等场景。

**档位表（动态增删行）**

每档包含：运算符（支持BETWEEN双值）、阈值（百分比模式下为百分比，支持正负值）、金额、条件备注、独立准入条件组。

---

#### 3.5.5 算子⑤ — 固定金额即发（fixed-reward）

**用途：** 满足条件就发固定金额，不排名不瓜分。支持多档配置——不同条件可发不同金额，无需重复添加多个组件。

**档位表（动态增删）：**

每档包含：

| 配置项 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| 发放条件 | `conditionGroup` | 条件组 | 该档位的准入条件，复用通用条件组 |
| 金额 | `amount` | 数值 | 满足该档条件时发放的金额 |
| 备注 | `note` | 文本(可选) | 该档位的说明，如"全勤底薪"、"基础底薪" |

> 💡 **匹配逻辑**：从上到下依次匹配，命中哪档发哪档金额。若需"只发最高档"则从高到低排列条件即可。

---

#### 3.5.6 算子⑥ — 固定奖池平分（pool-rule）

**用途：** 固定一笔总奖金，由达标人均分或按比例/加权分配，每人不超过封顶值。

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 准入条件 | `conditionGroup` | 条件组 | — | 复用通用条件组，决定哪些人参与瓜分 |
| 总奖池(元) | `pool` | 数字 | — | 该政策的总奖金金额 |
| 分配方式 | `dist` | 下拉 | `average` | `average`(平均) / `proportional`(按比例) / `weighted`(加权) |
| 单人上限(元) | `cap` | 数字 | — | 单人最多能分到的金额上限（留空不限） |

**分配方式说明**

| 值 | 说明 | 额外配置 |
|-----|------|---------|
| `average` | 所有达标人均分：每人 = 总奖池 ÷ 达标人数 | 无 |
| `proportional` | 按某指标占总量的比例分配：每人 = 奖池 × (该人指标值 / 所有人指标值之和) | `proportionalField`：选择比例指标（如场观UV、单月总场观） |
| `weighted` | 多维度加权后按权值比例分配：得分 = Σ(各指标占比 × 权重) | `weightDims`：维度数组，每个维度包含 `field`(指标) + `weight`(权重) |

---

#### 3.5.7 算子⑦ — 流水分成/返点（share-rule）

**用途：** 按绑定变量的阶梯比例计算分成金额。适用于"积分越多分成比例越高"的阶梯返点场景。

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 绑定变量 | `variable` | 下拉 | — | 用于计算分成的指标字段（从标准字段库选择，可引用计算字段定义生成的虚拟字段） |

**阶梯分成表（动态增删行）**：运算符 + 阈值 + 分成比例(%)

---

#### 3.5.8 算子⑧ — 裂变伯乐激励（referral-rule）

**用途：** 老主播推荐新主播加入的奖励机制。支持按人/笔固定奖金或阶梯表，可设个人封顶、总预算、名额限制。

> **注意：** 本组件不内置准入条件，准入逻辑统一通过前置的「条件筛选」组件（condition-filter）配置。

##### 基础配置

| 配置项 | 字段 | 类型 | 默认值 | 说明 |
|--------|------|------|--------|------|
| 奖励发给 | `rewardTarget` | 下拉 | `referrer` | `referrer`(推荐人) / `referee`(被推荐人) / `both`(双方均发) |
| 触发类型 | — | 固定 | `headcount` | 按推荐的有效新主播人数计算 |
| 奖励模式 | `rewardMode` | 下拉 | `per_unit` | `per_unit`(按人/笔固定奖金) / `tiered`(阶梯表) |
| 单人/笔奖(元) | `perUnit` | 数字 | — | per_unit模式时的每次奖金 |
| 个人封顶(元) | `cap` | 数字 | — | 单个推荐人最多能获得的奖金上限（留空不限） |
| 总预算封顶(元) | `totalBudget` | 数字 | — | 所有推荐人奖金总和上限（留空不限） |
| 全局名额 | `globalQuota` | 数字 | — | 伯乐激励总名额限制（留空不限） |
| 超限排序字段 | `overflowSort` | 下拉 | — | 超名额时排名依据字段（globalQuota有值时必填） |
| 超限排序方向 | `overflowOrder` | 下拉 | `desc` | `desc`(降序) / `asc`(升序) |

---

#### 3.5.9 算子⑨ — 表格直发（direct-rule）

**用途：** 按名单直接发放，不排名不瓜分。适用于审批后直接发放的场景。

> **注意：** 本组件不内置准入条件，如需准入筛选请使用前置的「条件筛选」组件（condition-filter）。

| 配置项 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| 字段配置 | `fields` | 数组 | 定义发放表格的列（仅可选择数字类型字段） |
| 已录入数据 | `entries` | 数组 | 已录入的发放名单数据 |

**字段配置（动态增删行）**：从系统字段列表中下拉选择，仅显示数字类型字段（含时长、天数、金额、积分、人数等关键词的字段）

### 3.6 审核管理（新增）

#### 3.6.1 功能说明

审核管理模块用于**部门负责人审核下属员工配置的活动政策**。审核环节处于"配置完成"与"生效执行"之间，确保每个上线的活动政策都经过负责人检查确认，避免配置错误导致的结算问题。

#### 3.6.2 审核流程

```
员工保存配置(草稿) → 提交审核 → 负责人收到审核通知 → 查看配置详情 → 通过(生效)/驳回(退回修改)
```

#### 3.6.3 提交审核（运营人员侧）

##### 入口

编辑页顶部操作栏，在"保存"按钮右侧新增「提交审核」按钮。

##### 提交前校验

提交审核时系统自动执行配置完整性校验：

| 校验项 | 规则 | 失败提示 |
|--------|------|---------|
| 活动名称 | 必须已选择活动 | "请先选择活动名称" |
| 政策配置 | 至少有1个政策包含至少1个算子节点 | "至少配置一条含算子的政策" |
| 数据导入 | 必须已导入数据文件 | "请先导入结算数据" |

##### 提交流程

1. 点击「提交审核」按钮
2. 系统执行校验，不通过则弹出错误提示
3. 校验通过后弹出提交确认弹窗：
   - 展示活动名称、政策数量、算子数量摘要
   - 选择审核人（下拉框，列出当前部门负责人列表）
   - 可填写提交说明（非必填）
4. 确认提交后：
   - 活动状态变为「待审核」
   - 配置内容**锁定不可编辑**
   - 审核人收到审核通知

##### 已驳回后修改

- 已驳回的活动，运营人员可查看驳回原因
- 点击「修改配置」按钮后恢复编辑状态（状态回到草稿）
- 修改完成后可重新提交审核

#### 3.6.4 审核列表页（部门负责人侧）

##### 入口

左侧导航新增「审核管理」菜单项（图标：`fa-check-circle`）。

##### 页面结构

**统计卡片区**

| 卡片 | 展示值 |
|------|--------|
| 待审核 | 待审核活动数量 |
| 已通过 | 本月已审核通过数量 |
| 已驳回 | 本月已驳回数量 |

**审核列表表格**

| 列名 | 说明 |
|------|------|
| 活动名称 | 提交审核的活动名称 |
| 提交人 | 提交审核的运营人员姓名 |
| 提交时间 | 提交审核的时间 |
| 政策数量 | 活动包含的政策条数 |
| 算子数量 | 所有政策的算子总数 |
| 审核状态 | 待审核 / 已通过 / 已驳回 |
| 操作 | 查看详情 / 审核 |

##### 交互规则

- 默认按提交时间倒序排列（最新提交在前）
- 支持按状态筛选（全部 / 待审核 / 已通过 / 已驳回）
- 待审核的记录操作列显示「审核」按钮（紫色高亮）
- 已审核的记录操作列显示「查看」按钮

#### 3.6.5 审核详情页（部门负责人侧）

##### 页面布局

审核详情页以**只读模式**完整展示活动配置内容，负责人可以逐项检查配置的合理性。

**顶部信息栏**

| 信息项 | 说明 |
|--------|------|
| 活动名称 | 被审核的活动名称 |
| 提交人 | 提交审核的运营人员 |
| 提交时间 | 提交审核的时间 |
| 审核状态 | 当前审核状态标签 |
| 提交说明 | 运营人员的提交备注（如有） |

**配置详情区域**（全部只读展示）

1. **活动基础信息**：活动名称、开始时间、结束时间
2. **数据导入信息**：已导入文件名、数据行数、字段映射情况
3. **政策画布概览**：每条政策的名称、合并关系、算子列表
4. **算子配置详情**：每个算子的完整配置参数（展开/收起）

**底部审核操作栏**（固定在页面底部）

| 按钮 | 样式 | 说明 |
|------|------|------|
| ✅ 审核通过 | 绿色主按钮 | 点击后弹出确认弹窗，根据达人确认开关走不同流程 |
| ❌ 驳回 | 红色按钮 | 点击后展开驳回原因输入框（必填），确认后活动状态变为"已驳回" |

##### 审核通过流程

1. 点击「审核通过」
2. 弹出确认弹窗
3. 确认后根据活动基础信息中「达人确认」开关分支处理：

**开启达人确认时：**
   - 活动状态变为「待达人确认」
   - 系统向相关达人推送小程序确认通知
   - 运营人员收到"审核通过，已推送达人确认"通知

**关闭达人确认时：**
   - 活动状态变为「财务内审中」
   - 系统自动推送财务内审
   - 运营人员收到"审核通过，已推送财务内审"通知

##### 审核驳回流程

1. 点击「驳回」
2. 展开驳回原因文本框（必填，最少10个字）
3. 点击「确认驳回」后：
   - 活动状态变为「已驳回」
   - 运营人员收到驳回通知 + 驳回原因
   - 运营人员可查看驳回原因并修改后重新提交

#### 3.6.6 审核历史记录

每次审核操作都会记录审核日志：

| 字段 | 说明 |
|------|------|
| 操作时间 | 审核操作的时间戳 |
| 操作人 | 执行审核的负责人姓名 |
| 操作类型 | 提交审核 / 上级审核通过 / 上级审核驳回 / 达人确认 / 财务内审通过 / 财务内审驳回 / 票据提交 / 推送FPP |
| 审核意见 | 驳回原因或通过备注 |

审核历史以**时间线**形式展示在审核详情页侧边栏，支持查看完整审核流程。

#### 3.6.7 业务规则

- 只有「草稿」和「已驳回」状态的活动可以提交审核
- 部门负责人只能审核**自己部门下属员工**提交的活动
- 审核中的活动配置锁定，运营人员不可编辑
- 审核驳回后，运营人员修改配置需重新走完整审核流程
- 同一活动可被多次驳回和重新提交，所有记录保留
- 审核操作不可撤销（通过/驳回后不可反悔）

### 3.7 结算单系统

#### 3.7.1 结算任务大盘

| 功能 | 说明 |

### 3.8 达人确认（复用现有小程序确认功能）

当活动「达人确认」开关开启时，上级审核通过后系统自动向达人推送小程序确认通知。达人在小程序中确认结算金额，所有达人确认完成后自动推送财务内审。确认截止时间取活动基础信息中的「确认/票据截止时间」。

---

### 3.9 财务内审（现有逻辑，不必修改）

复用现有财务审核流程。触发条件：
- 开启达人确认的活动：所有达人确认完成后自动推送
- 关闭达人确认的活动：上级审核通过后直接推送

内审通过 → 活动进入「待上传票据」状态，通知达人上传票据。内审驳回 → 活动退回「已驳回」状态，运营修改后重新提交上级审核。

### 3.10 票据管理

#### 3.10.1 功能说明

财务内审通过后，达人需通过小程序上传**发票**和**完税证明**。票据齐全后系统自动推送 FPP 进行付款。

#### 3.10.2 达人小程序票据上传页

##### 入口

达人收到小程序消息通知："您的结算已通过财务审核，请上传票据"，点击进入上传页面。

##### 页面内容

| 信息项 | 说明 |
|--------|------|
| 活动名称 | 当前结算的活动名称 |
| 结算金额 | 本次结算金额 |
| 上传截止时间 | 来自活动基础信息中配置的「确认/票据截止时间」 |

##### 票据上传区域

| 票据类型 | 说明 | 状态标识 |
|---------|------|---------|
| **发票** | 支持图片/PDF上传，可拍照或从相册选择 | 未上传 / 已上传 |
| **完税证明** | 支持图片/PDF上传，可拍照或从相册选择 | 未上传 / 已上传 |

- 每种票据支持上传多张（最多10张）
- 上传后可预览、删除重新上传
- 两种票据均上传后，底部显示「提交票据」按钮

#### 3.10.3 运营/财务侧票据查看

##### 结算明细中的票据状态

在结算明细表中，每行达人新增「票据状态」列：

| 票据状态 | 显示 | 说明 |
|---------|------|------|
| 未上传 | 灰色文字"未上传" | 达人尚未上传任何票据 |
| 部分上传 | 黄色标签 | 仅上传了发票或完税证明其中之一 |
| 票据齐全 | 按钮「查看票据」 | 发票和完税证明均已上传，点击可查看 |

- 点击「查看票据」按钮弹出票据预览弹窗，可查看/下载达人上传的发票和完税证明
- 支持按票据状态筛选达人列表（全部 / 未上传 / 部分上传 / 票据齐全）

#### 3.10.4 推送 FPP

当活动下**所有达人票据齐全**后：
1. 系统自动触发推送 FPP
2. 活动状态变为「已推送FPP」
3. 运营人员收到"票据齐全，已推送FPP"通知
4. 后续付款流程走现有 FPP 逻辑

#### 3.10.5 超时处理

- 超过「确认/票据截止时间」仍未上传票据的达人，系统自动发送催促提醒
- 运营人员可在结算明细中查看票据上传进度，手动催促

#### 3.10.6 业务规则

- 票据上传环节仅在财务内审通过后开启
- 发票和完税证明均为必传项，缺一不可
- 达人提交票据后可在截止时间前重新上传（覆盖）
- 单个达人票据齐全不触发 FPP，需全部达人齐全后统一推送
- 票据文件保留存档，支持后续查询和下载

## 四、固定逻辑（不可配置，引擎侧处理）

以下规则为系统固定逻辑，不在配置界面开放，由引擎侧统一处理：

| 规则 | 说明 |
|------|------|
| **有效天数计算** | 默认当天开播时长≥1小时计为1个有效天；可通过政策属性的"有效天数定义"自定义 |
| **取整规则** | 过程中保留小数点后2位（向下截断），最终金额取整数位（向下取整） |
