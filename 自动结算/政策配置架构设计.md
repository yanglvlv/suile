# 政策配置组件架构设计文档

---

## 一、现有 1625.html 问题诊断

### 1. 运算符体系残缺
| 问题 | 现状 | 应该 |
|------|------|------|
| 比较运算符 | 完美准入线写死 `≥` 一种 | 需支持 `>`, `>=`, `<`, `<=`, `=`, `!=`, `BETWEEN`, `IN` |
| 逻辑运算符 | 宽容线用 HTML 标签硬编码 `必达(AND)` / `二选一(OR)` | 需可配置的 AND / OR / NOT 逻辑组，且支持无限嵌套 |

### 2. 条件组不可编辑
- 完美准入线：条件数量写死 1 行（只有一个 select + 一个 input），无法添加/删除/排序
- 宽容降级线：AND 组固定 1 条、OR 组固定 2 条，结构锁死

### 3. 政策数据与文档不匹配
| 政策 | 缺失内容 |
|------|----------|
| 政策1-天龙 | 阶梯排名只有 1 档示例，实际需要 7 档 |
| 政策2-FC ONLINE | 完全缺少赛道二（ACU激励）和赛道三（伯乐激励） |
| 政策3-DNF随乐版 | 缺少多维加权权重配置（UV:50%/时长:30%/流水:20%）；缺少活跃底薪赛道 |
| 政策4-终极角落 | 缺少赛道三（CPT保底 取最高互斥） |
| 政策5-石器时代 | 缺少阶梯分成比例配置（30%-50%）；缺少生态奖励赛道 |
| 政策6-DNF机构版 | 缺少 S/B 级分层逻辑；缺少机构总榜赛道 |

### 4. 组件能力缺口
| 缺失能力 | 涉及政策 |
|-----------|----------|
| 条件组嵌套（AND 里嵌 OR） | 政策1 宽容降级线 |
| 多维加权评分 + 权重配置 | 政策3 |
| 取最高互斥合并关系 | 政策4 赛道三 |
| 动态变量/公式引用 | 政策5 影响力积分 |
| 机构分级嵌套（S/A/B/C 级别 × 各自阶梯） | 政策6 |
| 固定金额即发（达标即发 N 元） | 政策3 活跃底薪、政策4 基础激励 |
| 溢出比例计算 | 补充场景 |
| 阶梯分成（不同区间不同比例） | 政策5 影响力分成 |
| 超限降级排序 | 政策1（人数超过档位名额时按字段排序决定谁降级） |

## 二、新架构设计

### 核心设计原则
1. **一切皆条件组** — 所有筛选/准入/降级/分级逻辑统一用「条件组」表达
2. **运算符全覆盖** — 比较运算符 8 种，逻辑运算符 3 种（AND/OR/NOT）
3. **无限嵌套** — 条件组可以递归嵌套子条件组
4. **一切皆可增删** — 条件行、档位行、赛道 Tab 全部支持动态增删
5. **数据驱动渲染** — 所有 UI 由 JSON 数据模型驱动，改数据即改界面

### 组件树（8 种算子组件）

```
政策配置
├── 活动基础信息（名称、活动周期）
├── 数据导入 & 字段映射
└── 赛道画布（N 个赛道 Tab）
    └── 赛道属性
    │   ├── 赛道名称
    │   ├── 活动时间（每个赛道可独立设定，留空使用活动整体时间）
    │   ├── 有效天数定义（N小时=1有效天，留空使用全局默认）
    │   └── 合并关系
    │       ├── 独立核算（这条赛道自己算自己的）
    │       ├── 叠加（这条赛道的钱 + 其他赛道的钱）
    │       └── 取最高（跟其他赛道比，哪个多拿哪个）
    └── 算子节点（可拖拽排序）
        ├── ① 准入/筛选条件（设置谁有资格参与）
        │   ├── 条件组（AND/OR/NOT 可嵌套）
        │   │   └── 条件行（字段 + 运算符 + 值）
        │   ├── 未达标行为（淘汰/折算/降级）
        │   └── 折算公式配置（分母基准）
        │
        ├── ② 阶梯排名发奖
        │   ├── 排名方式（单指标/多维加权/复合公式）
        │   ├── [多维加权] 权重配置（N 个维度 + 权重百分比）
        │   ├── [多维加权] 惩罚规则（某维度未达标 → 折算比例）
        │   ├── 档位表（动态行：档名/门槛运算符/门槛值/金额或公式/名额）
        │   └── 超限降级规则（排序字段 + 排序方向）
        │
        ├── ③ 阶梯达标 / 溢出即发
        │   ├── 指标形态（绝对值/相对溢出比例/增长率）
        │   ├── 基线配置（基线字段 + 基线值）
        │   └── 档位表（动态行：门槛运算符/门槛值/金额或公式）
        │
        ├── ④ 固定金额即发（达标发 N 元）
        │   ├── 准入条件组（复用条件组）
        │   └── 发放金额（固定值/公式）
        │
        ├── ⑤ 固定奖池平分
        │   ├── 奖池金额
        │   ├── 分配方式（平均/按比例/加权）
        │   └── 单人上限
        │
        ├── ⑥ 流水分成 / 返点（阶梯版）
        │   ├── 绑定变量（可选标准字段或自定义公式）
        │   └── 阶梯分成表（动态行：区间下限运算符/值/分成比例）
        │
        └── ⑦ 裂变伯乐激励
            ├── 激励触发方式（带动人数/流水贡献/复合指标）
            ├── 单人奖励 / 阶梯奖励表
            └── 封顶上限

        └── ⑧ 多周期奖池瓜分
            ├── 瓜分方式（达标人数均分/按指标加权分配）
            ├── 取整规则（舍去小数/四舍五入/向上取整）
            ├── 瓜分说明备注
            └── 考核周期列表（动态增删）
                ├── 日期范围 + 天数
                ├── 达标条件组（复用通用条件组）
                ├── 总奖池金额
                └── 个人封顶金额
```

### 已删除的模块
- ~~全局防超发参数~~（生态系数、取整方式、小额拦截均已删除）
- ~~活动级结算频次~~（月结算/周结算/日结算 — 不需要在配置界面开放）
- ~~活动级结算维度~~（个人主播/机构聚合 — 不需要在配置界面开放）
- ~~赛道属性-数据维度~~（与结算维度重复，已删除）

### 固定逻辑（不可配置，引擎侧处理）
- **有效天数**：默认当天开播时长 ≥ 1小时计为1个有效天数；可在赛道属性中自定义（如：2小时=1天）
- **取整规则**：过程中保留小数点后2位（向下截断），最终金额取整数位（向下取整）

### 通用条件组数据模型（核心）

```json
{
  "logic": "AND",           // "AND" | "OR" | "NOT"
  "conditions": [
    {
      "type": "simple",     // 简单条件
      "field": "直播总时长(h)",
      "operator": ">=",     // ">", ">=", "<", "<=", "=", "!=", "BETWEEN", "IN"
      "value": 90
    },
    {
      "type": "simple",
      "field": "有效天数",
      "operator": ">=",
      "value": 15
    },
    {
      "type": "group",      // 嵌套子条件组
      "logic": "OR",
      "conditions": [
        { "type": "simple", "field": "直播总时长(h)", "operator": ">=", "value": 30 },
        { "type": "simple", "field": "有效天数", "operator": ">=", "value": 7 }
      ]
    }
  ]
}
```

### 比较运算符注册表

| 运算符 | 标签 | 值类型 | 说明 |
|--------|------|--------|------|
| `>` | 大于 | number | |
| `>=` | 大于等于 | number | |
| `<` | 小于 | number | |
| `<=` | 小于等于 | number | |
| `=` | 等于 | number/string | |
| `!=` | 不等于 | number/string | |
| `BETWEEN` | 区间 | [min, max] | 需要两个输入框 |
| `IN` | 属于列表 | string[] | 逗号分隔多值 |

### 逻辑运算符注册表

| 运算符 | 标签 | 说明 |
|--------|------|------|
| `AND` | 且 (全部满足) | 所有子条件都必须为真 |
| `OR` | 或 (任一满足) | 任一子条件为真即可 |
| `NOT` | 非 (取反) | 通常只包含一个子条件/组 |

### 算子节点完整数据模型

#### ① 准入/筛选条件
```json
{
  "type": "condition-filter",
  "name": "完美准入线",
  "conditionGroup": { /* 通用条件组 */ },
  "failAction": "eliminate",       // "eliminate" | "discount" | "degrade"
  "discountConfig": {              // failAction=discount 时生效
    "method": "bucket_min",        // "bucket_min" | "single_exempt" | "ratio" | "weighted_avg"
    "denominators": [              // 折算分母基准
      { "field": "直播总时长(h)", "target": 60 },
      { "field": "有效天数", "target": 15 }
    ]
  }
}
```

#### ② 阶梯排名发奖
```json
{
  "type": "rank-rule",
  "sortMethod": "single",         // "single" | "weighted" | "formula"
  "sortField": "场观UV",           // sortMethod=single 时
  "overflowSortField": "场观UV",   // 超限降级排序字段
  "overflowSortOrder": "desc",    // 超限降级排序方向 "desc" | "asc"
  "quotaOverflow": "none",        // "none" | "cascade" — 空额累积开关
  "quotaOverflowNote": "",        // 空额累积规则备注（可编辑）
  "tierMatchMode": "normal",      // "normal" | "min_dimension" — 档位匹配模式
  "weightConfig": [                // sortMethod=weighted 时
    { "field": "场观UV", "weight": 50 },
    { "field": "直播总时长(h)", "weight": 30 },
    { "field": "已执行总价", "weight": 20 }
  ],
  "penaltyRules": [                // 惩罚规则
    { "field": "已执行总价", "condition": "未完成", "action": "UV奖励按70%折算" }
  ],
  "tiers": [
    { "name": "第1档", "operator": ">=", "threshold": 40000, "amount": "2500", "limit": 1,
      "condGroup": { "logic": "AND", "conditions": [...] }  // 每档独立准入条件
    }
  ]
}
```

**空额累积说明：**
- `cascade`：某档位达标人数 < 限量名额时，空余名额向下一档累积
- `none`（默认）：不累积，各档位名额独立

**档位匹配模式说明（tierMatchMode）：**
- `normal`（默认）：逐档AND判断，每档 condGroup 中的所有条件必须全部满足才能入档
- `min_dimension`：取最低值降档模式——UV、时长、有效天数等各维度分别独立匹配能达到的最高档位，最终取其中最低的档位作为主播的实际档位。适用于"部分条件不满足时降档"的场景（如政策9：星座共鸣S2）

#### ③ 阶梯达标 / 溢出即发
```json
{
  "type": "target-rule",
  "metricType": "overflow_ratio",  // "absolute" | "overflow_ratio" | "growth_rate"
  "baselineField": "场观UV",
  "baselineValue": 10000,
  "tiers": [
    { "operator": ">=", "threshold": "100%", "amount": "2000" },
    { "operator": ">=", "threshold": "50%", "amount": "1500" }
  ]
}
```

#### ④ 固定金额即发
```json
{
  "type": "fixed-reward",
  "conditionGroup": { /* 通用条件组 */ },
  "amount": "300",                 // 固定值或公式
  "amountType": "fixed"            // "fixed" | "formula"
}
```

#### ⑤ 固定奖池平分
```json
{
  "type": "pool-rule",
  "pool": 2000,
  "distribution": "average",       // "average" | "proportional" | "weighted"
  "cap": 200,
  "minParticipants": 1
}
```

#### ⑥ 流水分成 / 返点（阶梯版）
```json
{
  "type": "share-rule",
  "variable": "影响力积分",
  "formula": "付费金额 / 10",       // 可选：变量计算公式
  "tiers": [
    { "operator": ">=", "threshold": 10000, "percentage": 50 },
    { "operator": ">=", "threshold": 5000, "percentage": 40 },
    { "operator": ">=", "threshold": 0, "percentage": 30 }
  ]
}
```

#### ⑦ 裂变伯乐激励
```json
{
  "type": "referral-rule",
  "triggerType": "headcount",      // "headcount" | "revenue" | "composite"
  "rewardMode": "per_unit",        // "per_unit" | "tiered"
  "perUnitAmount": 200,
  "tiers": [],                     // rewardMode=tiered 时使用
  "cap": 600
}
```

#### ⑧ 多周期奖池瓜分
```json
{
  "type": "multi-period-pool",
  "splitMethod": "equal_share",    // "equal_share" | "weighted_share"
  "splitNote": "按照达标主播人数瓜分",
  "roundRule": "floor",            // "floor" | "round" | "ceil"
  "periods": [
    {
      "dateRange": "11月18日-11月30日",
      "days": 13,
      "conditionGroup": { /* 通用条件组 — 达标条件 */ },
      "pool": 30000,               // 该周期总奖池(元)
      "cap": 500                    // 个人封顶(元)
    }
  ]
}
```

**说明：**
- 每个考核周期独立计算数据，满足条件的主播均分该周期的奖池
- 个人金额不超过封顶值
- 取整规则可选：舍去小数 / 四舍五入 / 向上取整

## 四、技术实现要点

1. **条件组递归渲染** — `renderConditionGroup(group, path)` 递归函数，path 用于定位嵌套层级
2. **运算符联动** — BETWEEN 时显示两个输入框，IN 时显示逗号分隔输入
3. **算子注册表** — `nodeRegistry` 对象，key 为算子类型，value 为 `{ render, getDefaultData, validate }`
4. **数据双向绑定** — 每个表单元素 onchange 时通过 path 更新 JSON 数据模型
5. **拖拽排序** — 算子节点和档位行都支持拖拽排序
6. **JSON 预览/导入** — 支持查看和导入整个政策的 JSON 配置
7. **超限降级** — 阶梯排名组件内置超限降级排序配置（排序字段 + 方向），当满足条件人数超过档位限额时按指定字段排名决定谁降级
8. **空额累积** — 阶梯排名组件内置空额累积开关（`quotaOverflow: cascade`），档位未满员时空余名额下推到下一档；支持自定义备注说明
9. **单维豁免折算** — 折算模式 `single_exempt`：仅看一个豁免维度（如时长），达标即全额(系数=1)，不达标按该维度折算；UI 自动收敛折算分母为单个豁免维度
10. **折算模式智能切换** — 从 `bucket_min` 切换到 `single_exempt` 时，自动从现有 `denoms[0]` 提取字段名和目标值作为豁免维度，避免数据丢失
11. **伯乐有效条件备注** — 裂变伯乐算子支持 `referralNote` 字段，可描述被推荐人有效条件、作废规则等业务说明
12. **多周期奖池瓜分** — 新增第8种算子 `multi-period-pool`，支持多个考核周期各自独立计算：每个周期有独立的日期范围、达标条件组、总奖池、个人封顶；瓜分方式支持达标人数均分和按指标加权分配
13. **有效天数定义可配置** — 赛道属性面板新增 `validDayHours` 字段，可自定义 N 小时 = 1个有效天（如 TF 政策为 2小时=1天）；留空或0时使用全局默认（1小时=1天）
14. **条件组路径解析增强** — `getByPath()` 函数升级，支持解析 `periods.0.conditionGroup` 等任意数组嵌套路径，不再局限于 `conditions` 数组
15. **赛道独立活动时间** — 赛道属性面板新增 `activityTime` 字段，每个赛道可独立设定活动时间（如"11月18日-1月06日"）；留空则使用活动整体时间
16. **档位独立条件（condNote）** — 排名算子每个 tier 支持 `condNote` 字段，用于表达每个档位不同的准入条件（如 T1-T6 要求有效天≥25+时长≥80h 而 T7-T9 只要有效天≥16+时长≥60h）；界面通过勾选"显示档位独立条件列"来切换显示
17. **排名算子增强字段** — `rankNote`（排名规则说明）和 `exclusionNote`（互斥排他说明）在排名算子中显示为提示卡片，用于描述特殊规则（如"本活动不与主播新势力叠加"）
